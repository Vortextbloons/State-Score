# Documentation Cleanup Skill

## Purpose

Clean and simplify an existing documentation set without expanding its scope.

Remove stale, duplicated, unnecessary, misleading, or poorly organized documentation while preserving useful and verified information.

This skill is not responsible for documenting new features or performing a complete documentation rewrite.

## Scope

Only modify documentation-related files, including:

* `README.md`
* `AGENTS.md`
* Files under `docs/`
* Documentation indexes
* Documentation generation configuration
* Documentation-specific scripts when required to keep generation working

Do not modify application code, tests, runtime configuration, APIs, or implementation behavior.

## Primary Objectives

* Remove duplicated information
* Remove obsolete documentation
* Remove filler and obvious implementation details
* Remove inaccurate file counts, line counts, and brittle references
* Merge documents that unnecessarily cover the same subject
* Split files only when a file has clearly unrelated responsibilities
* Shorten overly verbose sections
* Fix inconsistent terminology
* Fix broken internal links
* Remove links to files that no longer exist
* Remove empty, placeholder, abandoned, or autogenerated-looking sections
* Improve headings and document organization
* Preserve information that remains useful and verifiable

## Required Inspection

Before changing documentation:

1. Inspect the documentation tree.
2. Read `README.md`, `AGENTS.md`, and `docs/INDEX.md`.
3. Identify generated documentation such as `docs/ALL.md`.
4. Inspect the source files relevant to any claim being removed as stale or incorrect.
5. Inspect documentation generation scripts and relevant `package.json` commands.
6. Check Git history or references when the purpose of a document is unclear.

Do not assume documentation is obsolete merely because it is old.

## Cleanup Rules

### Remove

Remove content when it is:

* Repeated elsewhere with no additional value
* Contradicted by the current implementation
* About a removed feature, file, command, endpoint, or configuration option
* Generic filler that does not help a developer or operator
* A temporary update log that no longer provides useful historical context
* An exact line count or file-size claim likely to become stale
* An unnecessary source-file inventory
* A trivial explanation of self-evident code
* A placeholder, unfinished section, or abandoned plan
* An outdated migration note that no longer applies
* An unsupported claim that cannot be verified

### Merge

Merge documents when:

* They describe the same component or workflow
* One file contains only a small amount of unique information
* Readers must jump between multiple files to understand one subject
* Separate files cause duplicated maintenance

Preserve the clearest file path and update all links to it.

### Keep

Keep content when it documents:

* Architecture and trust boundaries
* Non-obvious behavior
* Public APIs and protocols
* Configuration users must provide
* Security or operational constraints
* Development, testing, deployment, or troubleshooting procedures
* Important design decisions
* Known limitations
* Commands that actually exist
* Information difficult to derive directly from the code

## Restrictions

* Do not add speculative documentation.
* Do not create new documentation simply to make the structure appear complete.
* Do not rewrite accurate sections without a concrete readability benefit.
* Do not introduce formatting churn.
* Do not change code behavior.
* Do not delete a document without checking incoming links.
* Do not manually edit generated aggregate files.
* Do not preserve duplicate content merely because it is technically accurate.
* Do not replace concise documentation with longer prose.
* Do not invent resolutions when documentation and code disagree; use the code as the source of truth.

## Generated Documentation

Treat generated files as build artifacts.

For this project:

* `docs/ALL.md` is generated from `docs/INDEX.md`.
* Never manually edit `docs/ALL.md`.
* Modify the source documentation or `docs/INDEX.md`.
* Regenerate the combined file after cleanup.

## Cleanup Workflow

1. Build a documentation inventory.
2. Mark each document as:

   * Keep
   * Shorten
   * Merge
   * Remove
   * Generated
   * Requires verification
3. Find duplicate sections and conflicting claims.
4. Verify questionable claims against implementation.
5. Make the smallest useful cleanup changes.
6. Update links and indexes after merges or removals.
7. Regenerate generated documentation.
8. Review the final diff for accidental content loss or formatting churn.

## Verification

Run the available documentation checks.

At minimum:

```bash
npm run combine-docs
```

Then verify:

* `docs/ALL.md` regenerated successfully
* `docs/ALL.md` was not manually edited
* All links in modified files resolve
* `docs/INDEX.md` references only existing files
* Documented commands exist in `package.json`
* Removed files have no remaining references
* Terminology is consistent
* No useful unique information was accidentally deleted
* The final documentation set is smaller or clearer for a defensible reason

Run tests, type checks, or builds only when documentation cleanup changes scripts, commands, paths, or generated output that depends on them.

## Parallel Work

Parallel subagents may be used for independent documentation areas.

Assign exclusive ownership of files or folders to avoid conflicting edits.

Useful cleanup assignments include:

* Duplicate-content analysis
* Stale implementation-reference analysis
* Link and index verification
* API and configuration consistency
* Architecture-document consolidation
* Final diff and deletion review

A final integration agent must review all deletions, merges, terminology changes, links, and generated output.

## Deletion Standard

Before deleting a file, confirm all of the following:

* Its useful information exists elsewhere or is no longer valid.
* It is not required by a documentation generator.
* No valid document links to it.
* No contributor workflow depends on it.
* Its removal improves clarity or reduces maintenance burden.

When uncertain, shorten or mark the questionable claim instead of deleting the entire document.

## Final Report

Provide one concise consolidated report containing:

* Files removed
* Files merged
* Files shortened
* Files intentionally left unchanged
* Duplicate or obsolete information removed
* Incorrect claims corrected
* Links or indexes repaired
* Commands and checks run
* Any cleanup that could not be completed safely
* Any failed checks and their reasons

Do not include repetitive per-agent reports.
